---
title: "Run GitHub Actions matrix jobs from changed file lists"
description: "Use the matrix or json inputs to produce JSON output that drives a GitHub Actions matrix strategy, spawning one parallel job per changed file."
---

GitHub Actions supports matrix strategies that expand a single job definition into multiple parallel runs, one per value in the matrix. The `changed-files` action can produce JSON-formatted output that feeds directly into a `matrix` strategy, letting you run a job for each changed file without hardcoding the list.

## The `matrix` input

Setting `matrix: true` is the easiest way to enable matrix-ready output. It is an alias that simultaneously sets `json: true` and `escape_json: false`, producing a JSON array that GitHub's `fromJSON()` function can consume without further processing.

```yaml
- name: Get changed files
  id: changed-files
  uses: tj-actions/changed-files@v47
  with:
    matrix: true
```

## The `json` input

If you need more control, use `json: true` directly. By default the JSON output is escaped so it can be passed safely through `${{ }}` expressions. Set `escape_json: false` when you need the raw JSON array.

```yaml
- name: Get changed files as JSON
  id: changed-files-json
  uses: tj-actions/changed-files@v47
  with:
    json: "true"
    escape_json: "false"
```

## Complete matrix workflow example

The workflow below is taken directly from the source repository. It runs in two phases: first a job collects the list of changed files and exports it as a matrix, then a second job fans out one run per file.

```yaml
name: Matrix Example

permissions:
  contents: read

on:
  workflow_dispatch:
  pull_request:
    branches:
      - main

jobs:
  changed-files:
    name: Get changed files
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.changed-files.outputs.all_changed_files }}
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - name: Get changed files
        id: changed-files
        uses: tj-actions/changed-files@v47
        with:
          matrix: true
      - name: List all changed files
        run: echo '${{ steps.changed-files.outputs.all_changed_files }}'

  matrix-job:
    name: Run Matrix Job
    runs-on: ubuntu-latest
    needs: [changed-files]
    strategy:
      matrix:
        files: ${{ fromJSON(needs.changed-files.outputs.matrix) }}
      max-parallel: 4
      fail-fast: false
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      - name: Test
        run: |
          echo ${{ matrix.files }}

  conditional-job:
    name: Run Conditional Job
    runs-on: ubuntu-latest
    needs: [changed-files]
    if: contains(needs.changed-files.outputs.matrix, 'README.md')
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      - name: Execute Conditional Logic
        run: |
          echo "README.md has been changed. Running conditional job."
```

### How this works

<Steps>
  <Step title="Collect changed files">
    The `changed-files` job runs `tj-actions/changed-files` with `matrix: true`. The `all_changed_files` output is a JSON array string like `["src/index.ts","README.md"]`.
  </Step>
  <Step title="Expose the array as a job output">
    The job-level `outputs` block re-exports the step output so it is accessible to downstream jobs via `needs.changed-files.outputs.matrix`.
  </Step>
  <Step title="Fan out with fromJSON()">
    The `matrix-job` uses `fromJSON(needs.changed-files.outputs.matrix)` in its `strategy.matrix` block. GitHub expands this into one job run per entry in the array.
  </Step>
  <Step title="Use the value in each run">
    Inside each matrix job run, `matrix.files` holds the path of the individual changed file.
  </Step>
</Steps>

## GitHub Actions matrix limits

<Warning>
GitHub Actions limits a matrix to **256 jobs** per workflow run. If your repository can have more than 256 files changed in a single push or pull request, consider filtering the `files` input to a narrower subset of paths before using matrix output.
</Warning>

The `max-parallel: 4` setting in the example above controls how many matrix jobs run concurrently. Adjust this value based on your runner capacity and resource constraints.
